# Plan 04-01: Archive System

**Phase:** 04
**Status:** In Progress
**Date:** 2026-03-17

## Goal
Implement auto-archive logic in Express. Two triggers:
1. Cold lead timer — stage 1 leads stale after `COLD_LEAD_DAYS` days, checked on server start and every 60 minutes.
2. Stage 9 (Complete) — immediate auto-archive on startup sweep (jobs already at stage 9 and not archived).

All archive events logged to `stage_events` table with `source = 'auto-archive'`.

## Requirements Addressed
- ARCH-01: Cold lead detection via `julianday` comparison in SQLite
- ARCH-02: COLD_LEAD_DAYS configurable via env var (default 7)
- ARCH-03: Stage 9 completed jobs auto-archived
- ARCH-04: Archive events logged to stage_events
- ARCH-11: runAutoArchive runs on server start + 60-min interval

## Tasks

### T1: Add COLD_LEAD_DAYS env var
- Append `COLD_LEAD_DAYS=7` to `.env.example` and `.env`

### T2: Create server/archive.js
- `runAutoArchive()` function
- Query stage 1 leads where `julianday('now') - julianday(stage_updated_at) >= COLD_LEAD_DAYS`
- Query stage 9 leads not yet archived
- ACID transactions for each batch update
- Log each archived person to `stage_events`

### T3: Wire into server/index.js
- Import `runAutoArchive` from `./archive.js`
- Call on startup after `seedJurisdictions()`
- Set 60-minute `setInterval`

### T4: Verify
- Insert a test person at stage 1 with `stage_updated_at` 8 days ago
- Restart server and confirm `is_archived = 1`

### T5: Write SUMMARY.md

## Notes
- Uses SQLite `julianday()` for date arithmetic — no external date library needed
- Transactions ensure partial failures don't leave inconsistent state
- Console log only fires when records are actually archived (no noise on clean runs)
