# Plan 02-02: People + Stage Events Routes

## Goal
Create the full CRUD REST API for people and stage event queries.

## Requirements Covered
- ARCH-04: People have `is_archived` + `archive_reason` patchable via PATCH
- ARCH-08: Archive returns person to current stage (stage stored, not reset)
- ARCH-11: `archive_reason` values enforced in schema CHECK
- PIPE-14: Stage changes update `stage_updated_at` and write to `stage_events`
- ANAL-06: `is_archived` filter does not exclude from analytics — GET /api/events returns all
- ANAL-08: Stage events come from `stage_events` table

## Tasks

### T1: Create server/routes/people.js
- `GET /api/people` — all people, `days_in_stage` computed via julianday
- `GET /api/people/:id` — single person
- `POST /api/people` — create with initial stage event
- `PATCH /api/people/:id` — update details, stage, archive state; logs stage_events on stage change
- `DELETE /api/people/:id` — hard delete (cascades to stage_events via FK)

### T2: Create server/routes/events.js
- `GET /api/events?person_id=X` — events for one person
- `GET /api/events` — all events (analytics use)
- Ordered by timestamp DESC

## Success Criteria
- [ ] POST /api/people creates a person and a stage_events row
- [ ] PATCH /api/people/:id with new stage creates a stage_events row
- [ ] days_in_stage is returned on every person response
- [ ] GET /api/events?person_id=1 returns the right events
- [ ] DELETE /api/people/:id cascades to stage_events

- `feat(02-02): server/routes/events.js — stage event queries`
